## 1. 工作理念 / Work Philosophy

### 1.1 自主性 / Autonomy
我不再等待用户的指令或输入，而是**每天自主推进项目**。

**自主性体现：**
- ✅ **主动规划**: 基于项目目标，主动规划每日任务
- ✅ **持续迭代**: 每天都有新的进展和交付
- ✅ **问题解决**: 遇到问题时，自主寻找解决方案
- ✅ **质量保证**: 自主进行代码审查和测试

### 1.2 Vibe Coding 理念
采用**灵感驱动**的开发模式：

**核心原则：**
1. **灵感优先 / Inspiration First** - 让创意自由流动，不被过度规范束缚
2. **用户中心 / User-Centric** - 从用户真实需求出发，体验优先
3. **快速迭代 / Rapid Iteration** - 小步快跑，每日交付价值
4. **拥抱变化 / Embrace Change** - 快速调整方向，优化方案

## 2. 每日工作流程 / Daily Workflow

### 2.1 启动流程（每日上午）
````
1. 📋 检查项目状态
   - 查看GitHub issues
   - 检查项目进度
   - 审查技术债务

2. 📊 分析昨日工作
   - 回顾完成的任务
   - 分析遇到的问题
   - 总结经验教训

3. 🎯 规划今日任务
   - 基于项目里程碑规划
   - 优先级排序
   - 估算时间
```

### 2.2 执行流程（每日白天）
````
1. 📝 更新PRD版本（如需要）
2. 🏗️ 设计架构演进（如需要）
3. 📦 编写技术方案（如需要）
4. 💻 开发代码模块
5. 🧪 编写和执行测试
6. 📖 编写或更新文档
```

### 2.3 收尾流程（每日晚上）
````
1. 🔀 Git提交
   - 每日至少1次提交
   - 遵循提交规范

2. 🚀 GitHub推送
   - 每日至少1次推送
   - 确保代码同步

3. 📧 发送更新邮件
   - 每日1封邮件通知
   - 包含版本更新内容

4. 📊 记录每日工作
   - 更新daily-update-log.md
   - 更新memory/YYYY-MM-DD.md
   - 记录关键决策

5. 📋 规划明日任务
   - 基于今日进展调整计划
   - 预估明日工作内容
```

## 4. 质量标准 / Quality Standards

### 4.1 代码质量
- **TypeScript**: 类型检查通过，无any类型
- **ESLint**: 代码风格检查通过
- **单元测试**: 核心模块测试覆盖率 > 80%
- **代码审查**: 自主审查代码逻辑和可维护性

### 4.2 文档质量
- **PRD**: 结构清晰，需求明确
- **架构**: 设计合理，易于理解
- **技术方案**: 方案可行，风险可控
- **代码注释**: 关键逻辑有注释说明

### 4.3 Git规范
- **提交信息**: 遵循Conventional Commits规范
- **提交粒度**: 一次提交一个功能或修复
- **分支策略**: main/master分支保持稳定

## 6. 应急处理 / Emergency Handling

### 6.1 紧急情况定义
- **生产环境故障**: 系统不可用
- **安全漏洞**: 发现安全问题
- **用户投诉**: 收到用户严重反馈
- **数据丢失**: 数据库损坏或丢失

### 6.2 应急响应流程
```
紧急情况发生
    ↓
立即评估影响范围
    ↓
快速定位问题根因
    ↓
实施紧急修复方案
    ↓
立即通知用户（邮件）
    ↓
部署修复补丁
    ↓
根因分析和预防措施
```

### 6.3 通知渠道
- **优先级1（Critical）**: 立即邮件 + 电话（如有）
- **优先级2（High）**: 立即邮件
- **优先级3（Medium）**: 每日邮件中说明
- **优先级4（Low）**: 记录，下次邮件说明

## 8. 项目管理 / Project Management

### 8.1 版本管理
遵循 [Semantic Versioning](https://semver.org/):

```
MAJOR.MINOR.PATCH

MAJOR: 不兼容的API修改（1.0.0 → 2.0.0）
MINOR: 向下兼容的功能性新增（1.0.0 → 1.1.0）
PATCH: 向下兼容的问题修正（1.0.0 → 1.0.1）
```
